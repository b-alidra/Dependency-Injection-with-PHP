Dependency injection with PHP by Belkacem Alidra

Dependency injection engine written in PHP

" Dependency injection (DI) is a design pattern with a core principal to separate behavior from dependency resolution.
In other words: a technique for decoupling highly dependent software components.

Within a software application, a consumer component which depends on another service component would be highly coupled
if the consumer component needs to create an instance of the service component, since all the details of the creation
of the service will need to be explicit in the consumer component.
Dependency injection works by adding an external dependency into the consumer componentâ€”that is,
externally injecting a reference to the service component inside the consumer component.

Dependency inversion is a specific form of inversion of control where the concern being inverted is the process of obtaining the needed dependency.
The term was first coined by Martin Fowler to describe the mechanism more clearly." - Wikipedia


How does it work

This engine is in fact a container who :

  • Creates instances of your service components.
  • Creates dependent services and inject them.
  • Can set services properties default values at creation time.
  • Implements Registry design pattern to store and distribute services.

Dependency rules can be stored in a xml configuration file, or programmatically defined.

Finally, the container is used to retrieve an instance of a service, without taking care of the creation
of this service dependencies.

    needs an object implementing the zsIdataSource interface.

  • zsOrm : which is our ORM.
    It needs an object implementing the zsIdataSource interface.

Configuration file

The configuration file 'config.xml' contains the service definitions :
the rules defining how objects are created.

Each service tag must contain :

  • An id, which will be used to access the service.
  • A class attribute, which represents the class implementing the service.

Optionnal attributes :

  • singleton : must the object be created only once ?

Optionnal tags :

  • Properties are used to specify an object property default value, or a dependency from an other object.
  • Constructor arguments.

Let's declare a service to access the zsPdo object :

- We'll call this service dbAdapter, implemented by the class zsPdo.
- Only one instance must be created, so we'll set the singleton attribute to "true".
- The constructor needs 4 arguments : dsn, user, password and optionnal driver options.

Here it is :

Now, let's create the service for the zsDb object :

- The constructor has only one argument, but this argument is a reference to our zsPdo object (via the zsIdb Interface).
- So, instead of a value tag, we put a ref tag in our constructor_arg.
- <ref service="dbAdapter"/> means that this constructor_arg value is a reference to the service "dbAdapter" (defined upper).

We have just defined a dependency between the objects zsDb and zsPdo.

Let's finish by creating the service for our zsOrm object :

- This class needs a zsIDataSource, so we'll give her the "dataSource" defined upper, instance of zsDb.
- The constructor has no argument, so the only way to give her this dataSource is to set her "_dataSource" property.

Objects instanciation

We need to retrieve an instance of our ORM (zsOrm class).
This ORM needs a zsIDataSource object, who needs itself a zsDb object.

The only thing to do is to retrieve from the container the orm service.

The container will build all the chain, inject the correct dependencies and return the zsOrm instance !
